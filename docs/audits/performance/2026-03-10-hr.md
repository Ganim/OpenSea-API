## Resumo Executivo

O módulo HR apresenta problemas de performance graves nas camadas de paginação e cache. Quatro dos dez critérios falharam, incluindo dois de severidade crítica. A paginação ocorre majoritariamente em memória (application-side) em vez de no banco de dados, o que significa que em produção com centenas de funcionários, ausências ou horas extras, a API busca todos os registros do banco antes de fatiar o resultado. Não há uso de Redis para nenhuma rota quente do módulo. No frontend, todos os hooks possuem `staleTime` corretamente configurado (5 min na maioria, 2 min para dados mutáveis), mas há ausência total de `React.memo` em itens de lista e de virtualização para listas potencialmente longas.

## Backend

### B1 — Ausência de N+1 Queries

**Resultado: WARN**

A maioria dos repositórios utiliza `include` corretamente em queries individuais e de listagem. Exemplos:

- `PrismaEmployeesRepository.findMany` (linha 410): `include: { user, department, position, supervisor }` — correto, uma query única com JOINs.
- `PrismaDepartmentsRepository.findMany` (linha 122): `include: { parent, manager, company }` — correto.
- `PrismaCompaniesRepository.findById` (linha 136): `include: { addresses, cnaes, fiscalSettings, stakeholders }` — correto.

**Problema identificado — N+1 real em `PrismaPayrollItemsRepository.createMany`:**

Arquivo: `src/repositories/hr/prisma/prisma-payroll-items-repository.ts`, linhas 34–43.

```typescript
async createMany(data: CreatePayrollItemSchema[]): Promise<PayrollItem[]> {
  const items: PayrollItem[] = [];
  for (const item of data) {
    const created = await this.create(item);  // await dentro de loop
    items.push(created);
  }
  return items;
}
```

Este método emite uma query `INSERT` separada para cada item da folha de pagamento, em vez de usar `prisma.payrollItem.createMany()`. Em uma folha com 50 funcionários e 10 itens cada, isso resulta em 500 queries sequenciais. O impacto é diretamente proporcional ao número de funcionários no momento do processamento da folha.

**Problema adicional — busca dupla em `update()` de vários repositórios:**

`PrismaPayrollItemsRepository.update` (linhas 185–188), `PrismaBonusesRepository.update` (linhas 161–162), `PrismaOvertimeRepository.update` (linhas 146–147), e `PrismaAbsencesRepository.update` (linhas 277–280) executam um `findUnique` seguido de `update`, gerando 2 queries onde 1 bastaria — o `update` do Prisma já lança erro se o registro não existe.

**Correção recomendada para `createMany`:**

```typescript
async createMany(data: CreatePayrollItemSchema[]): Promise<PayrollItem[]> {
  await prisma.payrollItem.createMany({
    data: data.map(item => ({
      payrollId: item.payrollId.toString(),
      employeeId: item.employeeId.toString(),
      type: item.type as PayrollItemType,
      description: item.description,
      amount: item.amount,
      isDeduction: item.isDeduction ?? false,
      referenceId: item.referenceId,
      referenceType: item.referenceType,
    })),
  });
  // Re-fetch para retornar os registros criados com IDs gerados
  return this.findMany({ payrollId: data[0].payrollId });
}
```

### B2 — Índices em Campos Frequentemente Consultados

**Resultado: PASS**

O schema apresenta cobertura de índices sólida em todos os modelos do módulo RH:

- `Employee`: `@@index([tenantId])`, `@@index([departmentId])`, `@@index([positionId])`, `@@index([supervisorId])`, `@@index([status])`, `@@index([hireDate])`, `@@index([deletedAt])`, `@@index([cpfHash])`, `@@index([pisHash])` — todos os campos de filtro cobertos.
- `Department`: `@@index([tenantId])`, `@@index([parentId])`, `@@index([managerId])`, `@@index([companyId])`, `@@index([isActive])`, `@@index([deletedAt])`.
- `Absence`: `@@index([tenantId])`, `@@index([employeeId])`, `@@index([type])`, `@@index([status])`, `@@index([startDate, endDate])`.
- `Payroll`: `@@index([tenantId])`, `@@index([status])`, `@@index([referenceYear, referenceMonth])`.
- `PayrollItem`: `@@index([payrollId])`, `@@index([employeeId])`, `@@index([type])`.
- `Bonus`: `@@index([tenantId])`, `@@index([employeeId])`, `@@index([date])`, `@@index([isPaid])`.
- `Deduction`: `@@index([tenantId])`, `@@index([employeeId])`, `@@index([isApplied])`, `@@index([isRecurring])`.
- `TimeEntry`: `@@index([employeeId, timestamp])` — índice composto correto para consultas de ponto por período.
- `Organization` (Supplier/Manufacturer): `@@index([tenantId])`, `@@index([type])`, `@@index([cnpjHash])`, `@@index([legalName])`, `@@index([status])`.

**Observação menor:** O modelo `WorkSchedule` possui apenas `@@index([tenantId])`. Se houver futuramente filtros por `isActive`, um índice adicional em `@@index([isActive])` seria benéfico.

### B3 — Paginação em Todos os Endpoints de Listagem

**Resultado: FAIL** (severidade: crítica)

Apenas dois endpoints realizam paginação no banco de dados. Os demais buscam todos os registros em memória e aplicam `.slice()` na camada de aplicação.

**Endpoints com paginação real no banco (CORRETO):**

| Endpoint                  | Repositório                                 | Método              |
| ------------------------- | ------------------------------------------- | ------------------- |
| `GET /v1/hr/employees`    | `PrismaEmployeesRepository`                 | —                   |
| `GET /v1/hr/departments`  | `PrismaDepartmentsRepository.findMany`      | `skip/take` correto |
| `GET /v1/hr/positions`    | `PrismaPositionsRepository.findMany`        | `skip/take` correto |
| `GET /v1/hr/companies`    | `PrismaCompaniesRepository.findMany`        | `skip/take` correto |
| `GET /v1/admin/suppliers` | `PrismaBaseOrganizationRepository.findMany` | `skip/take` correto |

**Endpoints com paginação em memória (FALHA CRÍTICA):**

**1. `GET /v1/hr/employees` — `ListEmployeesUseCase` (parcialmente correto):**

Arquivo: `src/use-cases/hr/employees/list-employees.ts`, linhas 48–113.

O use case chama `findMany`, `findManyByStatus`, `findManyByDepartment`, etc. — cada um retorna **todos** os registros do tenant — e depois aplica `.slice()` para paginar (linhas 109–113):

```typescript
const paginatedEmployees = employees.slice(startIndex, endIndex);
```

O repositório `findMany` (linha 410 do repositório) não aceita `skip/take`. Com 500 funcionários, a API carrega 500 objetos completos (incluindo dados cifrados) na memória antes de devolver 20. O decriptador de campos sensíveis é executado 500 vezes quando apenas 20 são necessários.

**2. `GET /v1/hr/absences` — `listAbsencesController`:**

Arquivo: `src/http/controllers/hr/absences/v1-list-absences.controller.ts`, linhas 41–53.

```typescript
// Simple pagination (in production, this should be handled by the repository)
const total = absences.length;
const start = (page - 1) * perPage;
const paginatedAbsences = absences.slice(start, start + perPage);
```

O comentário no código (`in production, this should be handled by the repository`) confirma que esta é uma implementação temporária jamais revisada.

**3. `GET /v1/hr/bonuses` — `listBonusesController`:**

Arquivo: `src/http/controllers/hr/bonuses/v1-list-bonuses.controller.ts`. A resposta retorna diretamente o array sem meta de paginação:

```typescript
return reply.status(200).send({ bonuses: bonuses.map(bonusToDTO) });
```

**4. `GET /v1/hr/deductions` — `listDeductionsController`:**

Idêntico ao padrão de bonuses — sem paginação real, sem meta de paginação na resposta.

**5. `GET /v1/hr/payrolls` — `listPayrollsController`:**

Retorna array completo sem paginação. A `findMany` do repositório inclui `items: true` para cada folha, tornando o payload potencialmente muito grande.

**6. `GET /v1/hr/overtime` — `listOvertimeController`:**

O use case de overtime aplica paginação em memória de forma similar ao de employees.

**Impacto estimado:** Em um tenant com 200 funcionários com 12 folhas de pagamento (cada uma com ~200 itens), a query `findManyByYear` no repositório de payrolls traz 12 objetos com 200 itens cada (2.400 `PayrollItem`) em uma única chamada sem limite.

### B5 — Select/Projeção para Limitar Campos

**Resultado: WARN**

A maioria das queries não utiliza `select` e retorna objetos completos do banco. O modelo `Employee` possui 40+ campos, incluindo dados bancários, endereço completo, documentos criptografados e metadados JSON.

**Observações:**

- `PrismaSuppliersRepository.create` (linha 30) usa `select: { typeSpecificData: true }` corretamente para buscar apenas o campo necessário ao calcular o próximo código sequencial.
- `PrismaManufacturersRepository.create` idem.

**Problema principal:** Todos os métodos `findMany` do repositório de funcionários retornam o objeto completo incluindo campos cifrados (CPF, PIS, conta bancária, chave PIX), que são decriptados um a um. Para listas, a maioria desses campos não é necessária — apenas `id`, `fullName`, `registrationNumber`, `status`, `departmentId`, `positionId` seriam suficientes.

**Sugestão:** Criar métodos `findManySummary` que usem `select` com apenas os campos necessários para a listagem, reservando o objeto completo (com decriptação) para o endpoint `GET /v1/hr/employees/:id`.

### F2 — React.memo em Itens de Lista Custosos

**Resultado: FAIL** (severidade: média)

Nenhum componente de item de lista no módulo HR usa `React.memo`. As funções de renderização de itens (`renderGridCard` e `renderListCard`) são definidas inline nas páginas, o que significa que são recriadas em cada re-render da página pai.

**Exemplo em `absences/page.tsx`, linhas 280–428:**

```typescript
const renderGridCard = (item: Absence, isSelected: boolean) => (
  <EntityContextMenu ...>
    <EntityCard ... />
  </EntityContextMenu>
);
```

Estas funções são passadas como props para `EntityGrid`. Cada atualização de estado (ex: abrir/fechar modal, mudança de filtro) recria as funções e força re-render de todos os itens visíveis.

**Padrão recomendado:**

```typescript
const AbsenceGridCard = React.memo(
  ({ item, isSelected, onApprove, onReject }: AbsenceCardProps) => (
    <EntityContextMenu ...>
      <EntityCard ... />
    </EntityContextMenu>
  )
);
```

O impacto é proporcional ao número de itens renderizados. Com `perPage: 50` (valor usado em `absences/page.tsx` linha 119), 50 re-renders desnecessários ocorrem em cada interação do usuário.

### F4 — Virtualização de Listas Longas

**Resultado: WARN**

Nenhum componente do módulo HR usa virtualização (`react-window`, `react-virtual`, `@tanstack/react-virtual` ou equivalente). A busca em toda a pasta `hr/` por essas bibliotecas não retornou resultados.

**Análise de risco:**

- `absences/page.tsx` busca até `perPage: 50` registros mas renderiza todos no DOM de uma vez via `EntityGrid`.
- `useListBonuses` e `useListDeductions` solicitam `perPage: 100` (linhas 39 e análogos), renderizando até 100 `EntityCard` simultaneamente.
- `useListCompanies`, `useHRAnalytics` e `work-schedules/page.tsx` solicitam `perPage: 100`.

**Avaliação mitigante:** O `EntityGrid` usa paginação no frontend (por meio de `totalItems` e controles de página), o que limita o número de itens renderizados em cada tela. Se o grid exibir apenas 20 itens por página, a virtualização é menos crítica. Sem acesso ao código de `EntityGrid`, não é possível confirmar esse comportamento.

**Recomendação:** Aplicar virtualização se `EntityGrid` renderizar todos os itens de uma vez. Para listas de funcionários (potencialmente 200+), a virtualização é especialmente importante.

## Plano de Ação Priorizado

### Prioridade 1 — Crítica (impacto imediato em produção)

**B3 — Paginação real no banco de dados**

1. Adicionar `page`, `perPage`, `skip`, `take` aos métodos `findMany`, `findManyByStatus`, `findManyByDepartment`, `findManyByPosition`, `findManyByCompany`, `findManyBySupervisor` no `PrismaEmployeesRepository`.
2. Refatorar `ListEmployeesUseCase` para delegar paginação ao repositório, removendo o `.slice()`.
3. Refatorar `PrismaAbsencesRepository.findMany` para aceitar `skip/take`.
4. Atualizar `listAbsencesController` para remover o comentário `// Simple pagination` e a lógica de `.slice()`.
5. Adicionar paginação nos repositórios de bonuses, deductions e overtime.
6. Atualizar os controllers de bonuses, deductions e payrolls para retornar `meta` de paginação.

### Prioridade 2 — Alta (afeta UX e tempo de carregamento inicial)

**F1 — Dynamic imports para modais**

Converter todos os modais de HR para importação dinâmica com `dynamic()` do Next.js. Estimar redução de 15–30% no tamanho do bundle inicial das páginas de HR.

**B1 — N+1 em createMany**

Substituir o loop sequencial em `PrismaPayrollItemsRepository.createMany` por `prisma.payrollItem.createMany()`.

### Prioridade 3 — Média (melhoria de escalabilidade)

**B4 — Cache Redis**

Implementar cache Redis (TTL 5 min) para `departments`, `positions`, `work-schedules` e `companies`. Reutilizar o padrão `L1/L2/L3` já presente em `PermissionService`.

**F2 — React.memo**

Extrair `renderGridCard` e `renderListCard` como componentes memoizados em todas as páginas de listagem do HR.

**B5 — Select/projeção**

Criar métodos `findManySummary` nos repositórios de employees que retornem apenas os campos necessários para listagem, evitando decriptação desnecessária de dados sensíveis.

### Prioridade 4 — Baixa (melhoria preventiva)

**F4 — Virtualização**

Verificar comportamento do `EntityGrid` e, se necessário, adicionar virtualização para listas com mais de 100 itens.

**B1 — Buscas duplas em update()**

Remover `findUnique` desnecessário antes de `update` nos repositórios de payroll items, bonuses, overtime e absences.
