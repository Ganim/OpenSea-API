# Auditoria de Performance - Modulo Stock

**Data**: 2026-03-10
**Escopo**: Backend (OpenSea-API) e Frontend (OpenSea-APP) - modulo `stock`
**Auditor**: Claude Opus 4.6

---

## Resumo Executivo

O modulo de estoque apresenta uma arquitetura funcional, mas com problemas criticos de paginacao em endpoints legados e oportunidades de melhoria em cache e projecao de dados. O frontend compensa parcialmente os problemas do backend com virtualizacao nas telas de listagem, mas mantem hooks legados que carregam dados sem paginacao.

**Nota Final: 6.5 / 10**

---

## Backend

### B1. Queries N+1 (Severidade: Alta)

**Resultado: WARN**

A maioria das queries utiliza `include` do Prisma corretamente para carregar relacoes em uma unica query. No entanto, existem dois problemas identificados:

1. **`detachItemsFromBins`** (`prisma-items-repository.ts`, linhas 657-683): Executa `N` queries individuais de `UPDATE` dentro de um loop `for...of`. Para cada item vinculado aos bins informados, uma query separada e executada. Com 100 itens, sao 101 queries (1 findMany + 100 updates).

   ```typescript
   // Problema: N+1 no loop de updates
   for (const item of items) {
     await prisma.item.update({
       where: { id: item.id },
       data: {
         lastKnownAddress: item.bin?.address ?? item.lastKnownAddress,
         binId: null,
       },
     });
   }
   ```

   **Correcao recomendada**: Usar `prisma.$transaction` com `updateMany` agrupado, ou construir um `CASE WHEN` via raw query para atualizar `lastKnownAddress` individualmente em batch.

2. **`listItemMovementsController`** (`v1-list-item-movements.controller.ts`, linhas 54-67): Faz uma query adicional ao `UserProfile` apos carregar os movimentos. Isso nao e N+1 (e uma unica query batch por `userId IN [...]`), mas adiciona uma round-trip ao banco desnecessariamente. O ideal seria incluir o user profile no `include` da query original do repositorio.

3. **`findAll` no `PrismaItemsRepository`** (linha 403): Carrega `variant -> product -> template + manufacturer` e `bin -> zone` para **todos** os itens. Isso gera JOINs profundos (4 niveis) em queries sem paginacao.

**Evidencias positivas**:

- `productInclude` centralizado no `PrismaProductsRepository` evita duplicacao.
- `findManyByProductWithAggregations` no repositorio de variantes usa `select` parcial no produto e agrega items com `select: { currentQuantity: true }` (boa pratica).

### B2. Indices em Campos Frequentemente Consultados (Severidade: Alta)

**Resultado: PASS**

O schema Prisma apresenta cobertura de indices robusta para os modelos de stock:

| Modelo       | Indices    | Observacao                                                                                                                                                                                                 |
| ------------ | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Product      | 14 indices | `tenantId`, `tenantId+status`, `status`, `fullCode`, `barcode`, `eanCode`, `upcCode`, `slug+deletedAt`, `name+deletedAt`, `templateId`, `supplierId`, `manufacturerId`, `organizationId`, `sequentialCode` |
| Variant      | 10 indices | `tenantId`, `productId`, `fullCode`, `barcode`, `eanCode`, `upcCode`, `slug+deletedAt`, `isActive`, `sequentialCode`                                                                                       |
| Item         | 12 indices | `tenantId`, `tenantId+status`, `variantId`, `binId`, `fullCode`, `barcode`, `eanCode`, `status`, `batchNumber`, `expiryDate`, `sequentialCode`, `expiryDate+deletedAt`, `batchNumber+variantId`            |
| ItemMovement | 8 indices  | `tenantId`, `itemId`, `userId`, `movementType`, `batchNumber`, `salesOrderId`, `itemId+createdAt`, `userId+movementType`, `createdAt DESC`                                                                 |
| Bin          | 8 indices  | `tenantId`, `zoneId`, `address`, `aisle`, `isActive`, `isBlocked`, `zoneId+aisle+shelf`                                                                                                                    |
| Warehouse    | 3 indices  | `tenantId`, `isActive`                                                                                                                                                                                     |
| Zone         | 4 indices  | `tenantId`, `warehouseId`, `isActive`                                                                                                                                                                      |

**Observacao menor**: O modelo `Variant` nao possui indice composto `[tenantId, deletedAt]`, diferente de `Product` e `Item` que o possuem. Todas as queries de variantes filtram por `tenantId` e `deletedAt = null`, entao esse indice composto seria benefico. Igualmente, `[tenantId, productId, deletedAt]` seria util para `findManyByProduct`.

### B3. Paginacao em Todos os Endpoints de Listagem (Severidade: Critica)

**Resultado: FAIL**

Este e o problema mais critico encontrado. Multiplos endpoints de listagem retornam **todos** os registros sem paginacao:

| Endpoint                              | Repositorio                      | Metodo                           | Paginado? |
| ------------------------------------- | -------------------------------- | -------------------------------- | --------- |
| `GET /v1/items`                       | `PrismaItemsRepository`          | `findAllWithRelations`           | **NAO**   |
| `GET /v1/items/by-product/:id`        | `PrismaItemsRepository`          | `findManyByProductWithRelations` | **NAO**   |
| `GET /v1/items?variantId=`            | `PrismaItemsRepository`          | `findManyByVariantWithRelations` | **NAO**   |
| `GET /v1/products` (legado)           | `PrismaProductsRepository`       | `findMany`                       | **NAO**   |
| `GET /v1/products?status=`            | `PrismaProductsRepository`       | `findManyByStatus`               | **NAO**   |
| `GET /v1/products?templateId=`        | `PrismaProductsRepository`       | `findManyByTemplate`             | **NAO**   |
| `GET /v1/products?manufacturerId=`    | `PrismaProductsRepository`       | `findManyByManufacturer`         | **NAO**   |
| `GET /v1/products?categoryId=`        | `PrismaProductsRepository`       | `findManyByCategory`             | **NAO**   |
| `GET /v1/variants` (legado)           | `PrismaVariantsRepository`       | `findMany`                       | **NAO**   |
| `GET /v1/item-movements`              | `PrismaItemMovementsRepository`  | `findAll` / `findManyBy*`        | **NAO**   |
| `GET /v1/purchase-orders?supplierId=` | `PrismaPurchaseOrdersRepository` | `findManyBySupplier`             | **SIM**   |

**Impacto**: Com centenas/milhares de produtos, variantes e itens, esses endpoints podem:

- Sobrecarregar a memoria do servidor Node.js
- Gerar payloads HTTP de varios MB
- Causar timeouts em conexoes lentas
- Travar a UI enquanto processa arrays grandes

**Nota**: Existem versoes paginadas (`useProductsPaginated`, `useItemsPaginated`, `useVariantsPaginated`) no frontend que chamam endpoints paginados separados (`list()` no service). Porem, os hooks legados (`useProducts`, `useItems`, `useVariants`) ainda sao utilizados ativamente em componentes como `product-workspace.tsx` e `overview/list/page.tsx`.

### B4. Cache Redis para Dados Frequentemente Acessados (Severidade: Media)

**Resultado: FAIL**

Nenhum endpoint do modulo stock utiliza cache Redis. As unicas referencias a cache no modulo sao:

- `generate-labels.ts`: Usa cache para templates de etiquetas (caso especifico)
- `suggest-address.ts`: Cache de sugestoes de endereco

Dados que se beneficiariam de cache Redis:

- **Templates**: Mudam raramente, sao carregados em quase toda operacao de produto/variante/item
- **Categorias**: Arvore de categorias raramente muda, carregada em listagens de produtos
- **Fabricantes**: Lista de fabricantes e estavel, carregada frequentemente
- **Tags**: Mudam raramente, carregadas com produtos

O `PermissionService` ja usa cache Redis L2 (conforme MEMORY.md). O mesmo padrao poderia ser aplicado para templates e categorias.

---

### B5. Select/Projecao para Limitar Campos de Resposta (Severidade: Media)

**Resultado: WARN**

O modulo apresenta um padrao misto:

**Pontos negativos**:

- `PrismaProductsRepository.findMany`: Carrega `template: true`, `supplier: true`, `manufacturer: true`, `variants` completas (com todos os campos), `productCategories` com `category`, e `productTags` com `tag`. Isso resulta em objetos de resposta muito grandes, especialmente quando variantes possuem atributos JSON extensos.
- `PrismaItemsRepository.findAll`: Carrega 4 niveis de relacoes (`item -> variant -> product -> template/manufacturer` + `item -> bin -> zone`) com todos os campos.
- `PrismaItemsRepository.findAllWithRelations`, `findManyByVariantWithRelations`, `findManyByProductWithRelations`, `findManyByBinWithRelations`: Todos repetem o mesmo `include` pesado sem usar `select`.

**Pontos positivos**:

- `findManyByProductWithAggregations` usa `select` parcial para produto (`fullCode`, `name`) e items (`currentQuantity`).
- `PrismaBinsRepository` nao carrega relacoes desnecessarias.

**Recomendacao**: Criar versoes "slim" dos metodos de listagem que usem `select` para retornar apenas os campos necessarios para tabelas/listas (id, nome, codigo, status). Manter os `include` completos apenas para endpoints de detalhe (`findById`).

## Frontend

### F1. Imports Dinamicos para Componentes Pesados (Severidade: Alta)

**Resultado: WARN**

Nenhum uso de `next/dynamic` ou `React.lazy` foi encontrado nas paginas do modulo stock. Todos os componentes sao importados estaticamente, incluindo:

- Modais de formulario (`variant-form-modal.tsx`, `item-entry-form-modal.tsx`)
- Editores de layout de zona (`layout-editor/`, `zone-map/`)
- Componentes de impressao/etiquetas

Esses componentes sao pesados (formularios com react-hook-form, editores visuais com drag-and-drop) e so sao renderizados mediante interacao do usuario, sendo candidatos ideais para `next/dynamic` com `ssr: false`.

**Excecao**: O `page.tsx` de labels usa `dynamic` para o componente de preview de impressao.

### F4. Virtualizacao para Listas Longas (Severidade: Media)

**Resultado: PASS**

As duas telas de maior volume de dados utilizam `@tanstack/react-virtual`:

## Plano de Acoes Prioritarias

### Prioridade 1 (Critica)

1. **Adicionar paginacao aos endpoints legados de items, products, variants e item-movements**. Usar o padrao `skip/take` do Prisma com `page/limit` na querystring e resposta `{ data, meta: { total, page, limit, pages } }`. O repositorio de purchase-orders ja implementa esse padrao.

2. **Depreciar os hooks legados** (`useProducts`, `useItems`, `useVariants`) e migrar todos os componentes para as versoes paginadas (`useProductsPaginated`, etc.).

### Prioridade 2 (Alta)

3. **Corrigir N+1 em `detachItemsFromBins`**: Substituir o loop por `updateMany` ou raw query batch.

4. **Adicionar imports dinamicos** para modais e editores pesados no modulo stock.

### Prioridade 3 (Media)

5. **Implementar cache Redis** para templates, categorias, fabricantes e tags (TTL 5-15 min).

6. **Configurar `staleTime`** nos hooks legados restantes e no `useItemMovements`.

7. **Criar versoes "slim"** dos metodos de repositorio para listagem, usando `select` em vez de `include` completo.

8. **Adicionar indice composto** `[tenantId, deletedAt]` e `[tenantId, productId, deletedAt]` no modelo `Variant`.
