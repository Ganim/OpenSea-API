## 3. Pipeline CI/CD

**Severidade**: ALTA
**Veredicto**: PASS

### OpenSea-API — 5 workflows

| Workflow                 | Trigger                      | Etapas                                                          |
| ------------------------ | ---------------------------- | --------------------------------------------------------------- |
| `ci.yml`                 | push main/develop, PR main   | Lint, Unit Tests + Coverage, Build                              |
| `fly-deploy.yml`         | push main                    | Deploy + Smoke Tests + Rollback automatico                      |
| `fly-deploy-staging.yml` | push develop                 | Deploy staging + Health check                                   |
| `security.yml`           | push main, PR, cron semanal  | npm audit, CodeQL, TruffleHog, Snyk (opt), License check, OWASP |
| `load-tests.yml`         | manual + repository_dispatch | k6 com 4 cenarios (smoke, auth, products, email)                |

**Pontos fortes**:

- Rollback automatico em falha de smoke test (producao)
- Ambiente staging separado com deploy automatico
- Scan de seguranca semanal (OWASP, CodeQL)
- Testes de carga com k6

**Lacuna identificada**: O CI **nao executa testes E2E** (`test:e2e`). Apenas testes unitarios sao executados no pipeline. Os testes E2E (285+) existem mas dependem de PostgreSQL + Redis, o que requer services no workflow. Recomenda-se adicionar um job E2E ao `ci.yml` com services PostgreSQL/Redis (similar ao `load-tests.yml`).

### OpenSea-APP — 4 workflows

| Workflow                 | Trigger            | Etapas                                      |
| ------------------------ | ------------------ | ------------------------------------------- |

| `fly-deploy.yml`         | push main          | Deploy Fly.io                               |
| `fly-deploy-staging.yml` | push develop       | Deploy staging                              |
| `lighthouse.yml`         | push main, PR main | Lighthouse CI (3 runs)                      |

**Pontos fortes**:

- Deploy de staging no branch `develop`

**Recomendacao**: Verificar no GitHub Settings que as seguintes regras estao ativas para `main`:

- Require pull request reviews (minimo 1)
- Require status checks to pass (lint, test-unit, build)
- Require branches to be up to date
- Do not allow force pushes
- Do not allow deletions
