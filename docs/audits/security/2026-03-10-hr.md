**Date:** 2026-03-10

## Summary

Foi auditado o módulo HR do OpenSea-API, cobrindo 19 sub-módulos: employees, departments, positions, companies, company-addresses, company-cnaes, company-fiscal-settings, company-stakeholder, suppliers, manufacturers, work-schedules, time-control, time-bank, absences, vacation-periods, overtime, payrolls, bonuses e deductions. A infraestrutura de segurança global (Helmet, CORS, JWT, rate limiting, Zod) está bem configurada. As falhas mais relevantes concentram-se na ausência de `verifyPermission` em endpoints sensíveis de time-control (clock-in/out, cálculo de horas), vacation-periods (schedule, cancel, start, complete, sell), e time-bank (get/list), além de duas quebras de isolamento multi-tenant onde `verifyTenant` foi omitido — sendo uma delas crítica na rota `GET /v1/hr/companies/:companyId/stakeholders`, que não valida se a company pertence ao tenant do usuário autenticado.

## Critical Issues (FAIL)

### [F-01] Ausência de `verifyPermission` em 15+ endpoints HR

**Critério:** #2 — RBAC permissions
**Severidade:** Critical

**Descrição:** Os seguintes endpoints protegem acesso apenas com `verifyJwt + verifyTenant`, sem nenhuma verificação de permissão RBAC. Qualquer usuário autenticado com um tenant selecionado pode acessar e modificar dados sensíveis de RH.

**Endpoints afetados:**

| Método | Rota                                   | Arquivo                                      | Impacto                                                        |
| ------ | -------------------------------------- | -------------------------------------------- | -------------------------------------------------------------- |
| POST   | `/v1/hr/time-control/clock-in`         | `v1-clock-in.controller.ts:16`               | Qualquer usuário pode registrar ponto por qualquer funcionário |
| POST   | `/v1/hr/time-control/clock-out`        | `v1-clock-out.controller.ts:16`              | Idem                                                           |
| POST   | `/v1/hr/time-control/calculate-hours`  | `v1-calculate-worked-hours.controller.ts:17` | Acesso a horas trabalhadas de qualquer funcionário             |
| GET    | `/v1/hr/time-control/entries`          | `v1-list-time-entries.controller.ts:17`      | Listagem irrestrita de registros de ponto                      |
| GET    | `/v1/hr/time-bank/:employeeId`         | `v1-get-time-bank.controller.ts:17`          | Leitura de banco de horas de qualquer funcionário              |
| GET    | `/v1/hr/time-bank`                     | `v1-list-time-banks.controller.ts`           | Listagem de banco de horas                                     |
| PATCH  | `/v1/hr/vacation-periods/:id/schedule` | `v1-schedule-vacation.controller.ts:21`      | Agendar férias sem autorização                                 |
| PATCH  | `/v1/hr/vacation-periods/:id/cancel`   | `v1-cancel-scheduled-vacation.controller.ts` | Cancelar férias sem autorização                                |
| PATCH  | `/v1/hr/vacation-periods/:id/start`    | `v1-start-vacation.controller.ts`            | Iniciar férias sem autorização                                 |
| PATCH  | `/v1/hr/vacation-periods/:id/complete` | `v1-complete-vacation.controller.ts`         | Concluir férias sem autorização                                |
| PATCH  | `/v1/hr/vacation-periods/:id/sell`     | `v1-sell-vacation-days.controller.ts`        | Vender dias de férias sem autorização                          |
| GET    | `/v1/hr/vacation-periods/:id`          | `v1-get-vacation-period.controller.ts:17`    | Leitura de períodos de férias                                  |
| GET    | `/v1/hr/vacation-periods`              | `v1-list-vacation-periods.controller.ts`     | Listagem de férias                                             |
| POST   | `/v1/hr/companies/check-cnpj`          | `v1-check-cnpj.controller.ts:18`             | Enumeração de CNPJs cadastrados                                |
| POST   | `/v1/hr/employees/check-cpf`           | `v1-check-cpf.controller.ts:13`              | Enumeração de CPFs cadastrados                                 |

**Impacto:** Um atacante com qualquer acesso ao tenant pode manipular dados de ponto, banco de horas e férias de todos os funcionários. Os endpoints de check-cpf/check-cnpj permitem enumeração de dados pessoais sensíveis.

**Correção recomendada:**

```typescript
// Exemplo para clock-in (aplicar padrão em todos os afetados)
preHandler: [
  verifyJwt,
  verifyTenant,
  createPermissionMiddleware({
    permissionCode: PermissionCodes.HR.TIME_CONTROL.CLOCK_IN,
    resource: 'time-control',
  }),
],
```

Os permission codes já existem no sistema RBAC. Verificar em `src/constants/rbac/permission-codes.ts` e criar os que estejam faltando para `time-control`, `time-bank` e `vacation-periods`.

## Warnings (WARN)

### [W-01] `verifyTenant` ausente em dois controllers de company-stakeholder

**Critério:** #1 — verifyJwt em todas as rotas
**Severidade:** Critical (dentro do WARN pois JWT está presente)

**Descrição:** O controller `v1-create-company-stakeholder.controller.ts:20-25` e `v1-get-company-stakeholders.controller.ts:14` omitem `verifyTenant` no `preHandler`. Sem `verifyTenant`, o campo `request.user.tenantId` não é garantido, e o use case pode operar sem isolamento de tenant — qualquer usuário autenticado sem tenant selecionado pode criar ou listar stakeholders de qualquer empresa via `companyId`.

```typescript
// v1-create-company-stakeholder.controller.ts — linha 20-25 (ATUAL, com falha)
preHandler: [
  verifyJwt,
  createPermissionMiddleware({ permissionCode: ... }),
]

// v1-get-company-stakeholders.controller.ts — linha 14 (ATUAL, com falha)
preHandler: [verifyJwt],
```

**Correção:**

```typescript
preHandler: [verifyJwt, verifyTenant, createPermissionMiddleware({ ... })],
```

### [W-02] Dados bancários e PII retornados sem granularidade por escopo

**Critério:** #9 — Segredos não expostos em respostas
**Severidade:** Critical (PII em resposta)

**Descrição:** O `employeeResponseSchema` (`src/http/schemas/hr/employees/employee.schema.ts:265-271`) expõe `cpf`, `pis`, `rg`, `bankCode`, `bankAgency`, `bankAccount`, `bankAccountType` e `pixKey` para qualquer usuário com permissão de leitura de funcionários. Não há distinção entre resposta de listagem (que não precisa de dados bancários) e resposta de detalhe (onde a exposição pode ser justificada com permissão específica).

**Impacto:** Violação de LGPD/GDPR — dados bancários e CPF de todos os funcionários ficam acessíveis a qualquer gestor com permissão `hr.employees.read`.

**Recomendação:**

- Criar `employeeListResponseSchema` sem campos bancários (`bankCode`, `bankAgency`, `bankAccount`, `bankAccountType`, `pixKey`) e sem `pis`.
- Manter `employeeResponseSchema` completo apenas para `GET /v1/hr/employees/:id` com permissão `hr.employees.read.sensitive` (ou escopo equivalente).
- Avaliar uso de `FIELD_ENCRYPTION_KEY` (já presente no env) para criptografar CPF e dados bancários no banco.

---

### [W-03] Campos de texto livre sem sanitização HTML explícita

**Critério:** #6 — Prevenção de XSS
**Severidade:** High

**Descrição:** Campos como `metadata` (`z.record(z.string(), z.unknown())`), `healthConditions[].description`, `emergencyContactInfo.name`, `reason` em ausências e `notes` em overtime aceitam strings arbitrárias. Embora respostas sejam JSON (não HTML), se qualquer parte do sistema usar esses dados em templates de email (EmailModule já integrado) ou relatórios PDF, o conteúdo pode causar XSS ou injeção em templates.

**Recomendação:**

- Adicionar `z.string().trim()` em campos de texto para evitar whitespace padding.
- Nos contextos de email e PDF, aplicar sanitização com `DOMPurify` (client) ou `sanitize-html` (server).
- Considerar schema de `metadata` mais restritivo (profundidade máxima, tipos de valor permitidos).
