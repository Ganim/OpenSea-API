## Resumo Executivo

O modulo Stock da OpenSea-API apresenta uma postura de seguranca **solida**, com autenticacao JWT, RBAC granular, validacao Zod e rate limiting aplicados de forma consistente. Foram identificados pontos de atencao menores em rate limiting por rota e sanitizacao de saida, mas nenhuma vulnerabilidade critica foi encontrada.

**Pontuacao Final: 9,2 / 10**

### 1. Todas as rotas utilizam middleware `verifyJwt` | PASS

**Severidade**: Critica
**Evidencia**: Todos os 106 arquivos de controller dentro de `src/http/controllers/stock/` possuem `verifyJwt` no array `preHandler` ou `onRequest`. Contagem confirmada: 212 ocorrencias de `verifyJwt` em 106 arquivos.

- `src/http/controllers/stock/products/v1-create-product.controller.ts:21` -- `preHandler: [verifyJwt, verifyTenant, ...]`
- `src/http/controllers/stock/products/v1-list-products.controller.ts:16` -- `preHandler: [verifyJwt, verifyTenant, ...]`
- `src/http/controllers/stock/warehouses/v1-create-warehouse.controller.ts:21` -- `preHandler: [verifyJwt, verifyTenant, ...]`

- `src/http/controllers/stock/address/v1-parse-address.controller.ts:14` -- `preHandler: [verifyJwt, ...]`

Adicionalmente, o middleware `verifyJwt` (`src/http/middlewares/rbac/verify-jwt.ts`) verifica nao apenas a assinatura do token, mas tambem se a sessao esta ativa, nao revogada e nao expirada (linhas 16-37). Isso e uma camada de defesa em profundidade exemplar.

### 3. Rate limiting em endpoints sensiveis | WARN

**Severidade**: Alta
**Evidencia**: Rate limiting global esta configurado em `src/app.ts:138-140` com 300 req/min por IP via `@fastify/rate-limit`. A configuracao detalhada esta em `src/config/rate-limits.ts` com 10 perfis diferentes (global, auth, public, query, mutation, heavy, admin, shareLink, emailSend, emailTest).

**Endpoints de autenticacao**: Protegidos com `rateLimitConfig.auth` (10 req/min) + `loginBruteforceGuard` que bloqueia IP apos 10 tentativas falhadas em 15 minutos (`src/http/controllers/core/auth/routes.ts:17-18`).

**Modulo Stock -- aplicacao parcial**: 8 dos 20 sub-modulos de rotas do Stock aplicam rate limiting por rota:

- `products/routes.ts` -- admin (300/min), mutation (100/min), query (120/min)
- `variants/routes.ts` -- mutation (100/min), query (120/min)
- `items/routes.ts` -- query (120/min), mutation (100/min)
- `labels/routes.ts`, `address/routes.ts`, `product-attachments/routes.ts`, `variant-attachments/routes.ts`, `product-care-instructions/routes.ts`

**Sub-modulos SEM rate limiting por rota** (dependem apenas do global 300/min):

- `warehouses/routes.ts`, `zones/routes.ts`, `bins/routes.ts`
- `categories/routes.ts`, `tags/routes.ts`, `templates/routes.ts`
- `manufacturers/routes.ts`, `suppliers/routes.ts`
- `volumes/routes.ts`, `purchase-orders/routes.ts`
- `care/routes.ts`, `item-movements/routes.ts`

O rate limiting global (300 req/min) e aplicado como fallback, o que mitiga o risco parcialmente.

**Recomendacao**: Aplicar `rateLimitConfig.mutation` e `rateLimitConfig.query` a todos os sub-modulos de rotas do Stock para uniformidade de defesa em profundidade.

**Resultado**: WARN

### 4. Validacao de entrada via Zod em TODAS as rotas | PASS

**Severidade**: Critica
**Evidencia**: O Fastify esta configurado com `fastify-type-provider-zod` (`src/app.ts:43-44`), e todos os controllers do Stock utilizam `withTypeProvider<ZodTypeProvider>()` com schemas Zod para `body`, `params` e `querystring`.

Schemas reutilizaveis centralizados em `src/http/schemas/stock/`:

- `products/product.schema.ts` -- `createProductSchema`, `updateProductSchema`, `productResponseSchema`
- `common.schema.ts` -- `idSchema` (UUID), `paginationSchema`, `nameSchema`, etc.

Exemplos de validacao:

- `createProductSchema` (`product.schema.ts:28-38`): nome com min/max, description max 1000 chars, status enum, templateId como UUID
- `listProductsController` (`v1-list-products.controller.ts:27-31`): querystring com `templateId`, `manufacturerId`, `categoryId` todos como `z.string().uuid().optional()`
- `parseAddressController` (`v1-parse-address.controller.ts:28`): `address: z.string().min(1).max(50)`

Schemas de resposta tambem sao validados (serializerCompiler), prevenindo vazamento de dados nao intencionais.

**Ponto de atencao menor**: O campo `attributes` nos schemas de produto usa `z.record(z.string(), z.any())`, o que aceita valores arbitrarios. Isso e intencional para atributos dinamicos baseados em templates, mas deve ser monitorado.

**Resultado**: PASS

### 5. Prevencao de SQL injection (queries parametrizadas via Prisma) | PASS

**Severidade**: Critica
**Evidencia**: O modulo Stock utiliza exclusivamente Prisma ORM para acesso a dados, com queries parametrizadas. As unicas queries raw no modulo Stock sao:

1. `src/repositories/stock/prisma/prisma-products-repository.ts:362` -- `$queryRaw` com tagged template literal (parametrizado):

   ```
   const result = await prisma.$queryRaw<[{ nextval: bigint }]>`...`
   ```

2. `src/repositories/stock/prisma/prisma-manufacturers-repository.ts:322` -- Mesmo padrao de sequence query.

Ambas usam **tagged template literals** (`$queryRaw\`...\``), que sao automaticamente parametrizadas pelo Prisma. Nao ha uso de `$queryRawUnsafe` no modulo Stock.

**Nota sobre outros modulos** (fora do escopo, mas relevante): O modulo Email (`prisma-email-messages-repository.ts:222-223`) e Finance (`prisma-finance-entries-repository.ts:423+`) usam `$queryRawUnsafe`, porem com parametros posicionais (`$1`, `$2`, etc.) passados como argumentos separados, o que previne SQL injection. Os valores de `groupBy` (day/week/month) sao derivados de uma condicao ternaria com valores fixos, nao de input do usuario.

**Resultado**: PASS

### 6. Prevencao de XSS (sanitizacao de saida) | WARN

**Severidade**: Alta
**Evidencia**: A API e primariamente uma API REST que retorna JSON, o que reduz significativamente o risco de XSS reflexivo. No entanto:

**Pontos positivos**:

- Helmet esta configurado com Content-Security-Policy (`src/app.ts:80-98`)
- O modulo Email possui `HtmlSanitizerService` (`src/services/email/html-sanitizer.service.ts`) para sanitizar corpos HTML
- Schemas de resposta Zod atuam como filtros de saida, prevenindo campos nao esperados

**Pontos de atencao**:

- Campos de texto livre no modulo Stock (`name`, `description`, `attributes`) nao passam por sanitizacao explicita de saida. Se consumidos por um frontend que renderize HTML sem escaping (ex: `dangerouslySetInnerHTML`), poderiam ser vetor de XSS armazenado.
- O campo `attributes` aceita `z.any()`, o que permite armazenar HTML/scripts arbitrarios.

**Mitigacao existente**: O frontend Next.js/React faz escaping automatico por padrao. A CSP configurada no Helmet tambem mitiga. O risco residual e baixo, mas uma camada de sanitizacao no backend (strip HTML tags de campos de texto) seria defesa em profundidade.

**Resultado**: WARN

## Recomendacoes de Melhoria

### Prioridade Alta

1. **Rate limiting por rota em todos os sub-modulos Stock**: Adicionar `rateLimitConfig.mutation` e `rateLimitConfig.query` a `warehouses/routes.ts`, `zones/routes.ts`, `bins/routes.ts`, `categories/routes.ts`, `tags/routes.ts`, `templates/routes.ts`, `manufacturers/routes.ts`, `suppliers/routes.ts`, `volumes/routes.ts`, `purchase-orders/routes.ts`, `care/routes.ts` e `item-movements/routes.ts`. Seguir o padrao ja usado em `products/routes.ts`.

### Prioridade Media

2. **Sanitizacao de campos de texto**: Considerar adicionar um schema transform Zod que remova tags HTML de campos `name`, `description` e similares. Exemplo:

   ```typescript
   const safeStringSchema = z
     .string()
     .transform((s) => s.replace(/<[^>]*>/g, ''));
   ```

3. **Restringir `z.any()` em `attributes`**: Substituir `z.record(z.string(), z.any())` por `z.record(z.string(), z.union([z.string(), z.number(), z.boolean(), z.null()]))` para prevenir armazenamento de objetos/HTML complexos.

### Prioridade Baixa

4. **CSP mais restritiva**: Avaliar remover `'unsafe-inline'` de `scriptSrc` e `styleSrc` quando Swagger UI puder ser servido com nonces.
