## 1. Transacoes em Operacoes Multi-Step [FAIL - CRITICO]

### 1.1 RegisterItemEntryUseCase - SEM TRANSACAO

**Arquivo**: `src/use-cases/stock/items/register-item-entry.ts`

Este use case executa **duas operacoes de escrita independentes** sem transacao:

1. `this.itemsRepository.create(...)` (linha 177)
2. `this.itemMovementsRepository.create(...)` (linha 202)

**Cenario de inconsistencia**: Se a criacao do item ocorrer com sucesso mas a criacao do `ItemMovement` falhar (por exemplo, por timeout de rede, violacao de constraint no userId, ou erro de banco), o sistema tera um item no inventario **sem registro de movimentacao correspondente**. Isso compromete a rastreabilidade do estoque e impossibilita auditorias.

**Agravante - Race condition no sequentialCode**: Entre as linhas 155-159, o `findLastByVariantId` e lido e o `nextSeq` e calculado. Se duas requisicoes concorrentes executarem simultaneamente para a mesma variante, ambas lerao o mesmo `lastItem.sequentialCode` e tentarao criar itens com o mesmo `sequentialCode` e `fullCode`. O `fullCode` tem constraint `@unique` no Prisma, entao uma delas falhara - mas a que falhar pode ja ter avancado a sequence do PostgreSQL.

### 1.2 RegisterItemExitUseCase - SEM TRANSACAO

**Arquivo**: `src/use-cases/stock/items/register-item-exit.ts`

Duas operacoes de escrita sequenciais sem transacao:

1. `this.itemsRepository.save(item)` (linha 86) - atualiza `currentQuantity`
2. `this.itemMovementsRepository.create(...)` (linha 89) - registra movimentacao

**Cenario de inconsistencia**: Se o `save` do item ocorrer com sucesso (reduzindo a quantidade) mas a criacao do `ItemMovement` falhar, o estoque ficara reduzido sem registro de movimentacao. Pior: a quantidade perdida nao pode ser rastreada nem revertida pelo sistema, pois nao ha registro de por que diminuiu.

### 1.3 TransferItemUseCase - SEM TRANSACAO

**Arquivo**: `src/use-cases/stock/items/transfer-item.ts`

Duas operacoes de escrita sequenciais sem transacao:

1. `this.itemsRepository.save(item)` (linha 85) - atualiza `binId` e `lastKnownAddress`
2. `this.itemMovementsRepository.create(...)` (linha 88) - registra movimentacao de transferencia

**Cenario de inconsistencia**: Item aparece no novo bin sem registro de transferencia. A trilha de auditoria fica incompleta.

### 1.4 ConfigureZoneStructureUseCase - SEM TRANSACAO

**Arquivo**: `src/use-cases/stock/zones/configure-zone-structure.ts`

Esta e a operacao mais complexa do modulo stock, executando ate **7 operacoes de escrita** em sequencia sem transacao:

1. `binsRepository.updateAddressMany(...)` (linha 155) - atualiza enderecos de bins preservados
2. `binsRepository.createMany(...)` (linha 161) - cria novos bins
3. `binsRepository.softDeleteMany(...)` (linha 171) - soft-delete de bins vazios
4. `itemsRepository.detachItemsFromBins(...)` (linha 201) - desvincula itens de bins ocupados
5. `itemMovementsRepository.createBatchForZoneReconfigure(...)` (linha 208) - registra movimentacoes
6. `binsRepository.softDeleteMany(...)` (linha 218) - soft-delete de bins bloqueados
7. `zonesRepository.updateStructure(...)` (linha 245) - atualiza estrutura da zona

**Cenario de inconsistencia**: Se a operacao falhar entre os passos 4 e 5, os itens estarao desvinculados dos bins mas sem registro de movimentacao, tornando impossivel rastrear de onde vieram. Se falhar apos o passo 6 mas antes do 7, os bins estarao deletados mas a estrutura da zona ainda reflete a configuracao antiga.

### 1.5 DeleteZoneUseCase - SEM TRANSACAO

**Arquivo**: `src/use-cases/stock/zones/delete-zone.ts`

Ate **4 operacoes de escrita** sem transacao quando `forceDeleteBins=true`:

1. `itemsRepository.detachItemsFromBins(...)` (linha 85)
2. `itemMovementsRepository.createBatchForZoneReconfigure(...)` (linha 92)
3. `binsRepository.deleteByZone(...)` (linha 101)
4. `zonesRepository.delete(...)` (linha 104)

### 1.6 ReorderCategoriesUseCase - SEM TRANSACAO

**Arquivo**: `src/use-cases/stock/categories/reorder-categories.ts`

Itera sobre N categorias fazendo update individual (linhas 22-38). Se falhar no meio, parte das categorias tera a nova ordem e parte tera a ordem antiga, deixando o display order inconsistente.

### 1.7 BatchTransferItemsUseCase - TRANSACAO OPCIONAL

**Arquivo**: `src/use-cases/stock/items/batch-transfer-items.ts`

Este use case e o **unico** no modulo stock que usa `TransactionManager`, porem de forma **opcional** (linhas 66-69):

```typescript

  return this.transactionManager.run(() => doTransfer());
}
return doTransfer();
```

Se a factory nao injetar o `TransactionManager`, a operacao executa sem transacao. E necessario verificar se a factory de fato injeta o `PrismaTransactionManager`.

### 1.8 Operacoes com Transacao (Positivo)

O repositorio `PrismaProductsRepository.create()` usa `prisma.$transaction()` quando `categoryIds` esta presente (linhas 58-74). O `PrismaProductsRepository.update()` tambem usa transacao para atualizar categorias (linhas 272-308). Estas sao as unicas transacoes no modulo stock.

## 2. Isolamento Multi-Tenant [FAIL - CRITICO]

### 2.1 VolumeRepository.getItemsByVolumeId - SEM FILTRO DE TENANT

**Arquivo**: `src/repositories/stock/prisma/prisma-volumes-repository.ts`, linhas 117-127

```typescript
async getItemsByVolumeId(volumeId: string): Promise<VolumeItem[]> {
  const items = await prisma.volumeItem.findMany({
    where: { volumeId },
  });
  return items.map((item) => VolumeItemMapper.toDomain(item));
}
```

A interface do repositorio (`volumes-repository.ts`) define `getItemsByVolumeId(volumeId: string)` **sem parametro `tenantId`**. Embora o volume em si seja validado com `tenantId` no use case antes de chamar este metodo, o modelo `VolumeItem` nao possui campo `tenantId` proprio - depende da relacao com `Volume`. Isto nao e uma falha de vazamento direta (o volumeId ja foi validado), mas viola o principio de defesa em profundidade.

### 2.2 VolumeRepository.countItemsByVolumeId - SEM FILTRO DE TENANT

**Arquivo**: `src/repositories/stock/prisma/prisma-volumes-repository.ts`, linhas 129-134

Mesmo problema do `getItemsByVolumeId`. Nao filtra por `tenantId`.

### 2.3 VolumeRepository.removeItem - SEM FILTRO DE TENANT

**Arquivo**: `src/repositories/stock/prisma/prisma-volumes-repository.ts`, linhas 108-115

```typescript
async removeItem(volumeId: string, itemId: string): Promise<void> {
  await prisma.volumeItem.deleteMany({
    where: { volumeId, itemId },
  });
}
```

Embora `volumeId` ja tenha sido validado com `tenantId` no use case, a operacao de delete nao filtra por tenant diretamente. Um atacante com conhecimento de um `volumeId` de outro tenant poderia potencialmente manipular itens.

### 2.4 VolumeRepository.delete - SEM FILTRO DE TENANT

**Arquivo**: `src/repositories/stock/prisma/prisma-volumes-repository.ts`, linhas 60-69

```typescript
async delete(id: string): Promise<void> {
  await prisma.volume.update({
    where: { id },
    data: { deletedAt: new Date() },
  });
}
```

O soft-delete e feito apenas por `id`, sem validacao de `tenantId`. O use case valida `tenantId` antes, mas a camada de repositorio nao garante isolamento por si so.

### 2.5 Product.getNextSequentialCode - GLOBAL (sem tenantId)

**Arquivo**: `src/repositories/stock/prisma/prisma-products-repository.ts`, linhas 361-366

```typescript
async getNextSequentialCode(): Promise<number> {
  const result = await prisma.$queryRaw<[{ nextval: bigint }]>`
    SELECT nextval(pg_get_serial_sequence('products', 'sequential_code'))
  `;
  return Number(result[0].nextval);
}
```

A sequence do PostgreSQL e **global** (compartilhada entre todos os tenants). Isso significa que:
- Os `sequentialCode` dos produtos nao sao sequenciais dentro de cada tenant (ex: tenant A recebe 1, 3, 5; tenant B recebe 2, 4, 6).
- Nao e um vazamento de dados, mas revela informacao sobre a atividade do sistema a um atacante observante.
- Os `fullCode` gerados contem gaps por tenant.

### 2.6 Manufacturer.getNextSequentialCode - GLOBAL (sem tenantId)

**Arquivo**: `src/repositories/stock/prisma/prisma-manufacturers-repository.ts`, linhas 321-325

Mesmo problema do item 2.5 - sequence global compartilhada entre tenants.

### 2.7 Operacoes de update/save sem verificacao de tenantId

Os seguintes metodos de repositorio executam operacoes de update usando apenas o `id` do registro, sem filtrar por `tenantId`:

- `PrismaItemsRepository.update()` (linha 605)
- `PrismaItemsRepository.save()` (linha 627)
- `PrismaVariantsRepository.update()` (linha 378)
- `PrismaVariantsRepository.save()` (linha 414)
- `PrismaVariantsRepository.delete()` (linha 450)
- `PrismaProductsRepository.update()` (linha 313)
- `PrismaProductsRepository.save()` (linha 332)
- `PrismaProductsRepository.delete()` (linha 351)
- `PrismaBinsRepository.update()` (linha 290)
- `PrismaBinsRepository.save()` (linha 300)
- `PrismaBinsRepository.delete()` (linha 316)
- `PrismaZonesRepository.update()` (linha 196)
- `PrismaZonesRepository.updateStructure()` (linha 207)
- `PrismaZonesRepository.updateLayout()` (linha 219)
- `PrismaZonesRepository.save()` (linha 234)

**Atenuante**: Todos os use cases verificam a existencia do recurso com `tenantId` antes de chamar o update/save/delete. Portanto, um uso correto do sistema nao permite vazamento cross-tenant. Porem, a camada de repositorio **nao e segura por si so** - qualquer novo use case que chame estes metodos sem validacao previa pode causar vazamento.

**Recomendacao**: Adicionar `tenantId` como parametro obrigatorio nos metodos `update`, `save` e `delete` de todos os repositorios, e incluir na clausula `where` do Prisma.

## 4. Cascade Deletes no Schema Prisma [WARN]

### 4.1 Cascades Corretos (Positivo)

As seguintes relacoes usam `onDelete: Cascade` apropriadamente:

- `ProductCategory.product` -> `Product` (Cascade) -- correto, tabela de juncao
- `ProductCategory.category` -> `Category` (Cascade) -- correto, tabela de juncao
- `ProductTag.product` -> `Product` (Cascade) -- correto, tabela de juncao
- `ProductTag.tag` -> `Tag` (Cascade) -- correto, tabela de juncao
- `VariantPriceHistory.variant` -> `Variant` (Cascade) -- correto, historico
- `VariantImage.variant` -> `Variant` (Cascade) -- correto, dependente
- `ProductCareInstruction.product` -> `Product` (Cascade) -- correto, dependente
- `ProductAttachment.product` -> `Product` (Cascade) -- correto, dependente
- `VariantAttachment.variant` -> `Variant` (Cascade) -- correto, dependente
- `VolumeItem.volume` -> `Volume` (Cascade) -- correto, dependente
- `Zone.warehouse` -> `Warehouse` (Cascade) -- correto, dependente
- `Bin.zone` -> `Zone` (Cascade) -- correto, dependente
- `PurchaseOrderItem.order` -> `PurchaseOrder` (Cascade) -- correto, dependente
- `Variant.product` -> `Product` (Cascade) -- correto, dependente
- `UnitConversion.variant` -> `Variant` (Cascade) -- correto, dependente
- `VariantPromotion.variant` -> `Variant` (Cascade) -- correto, dependente
- `VariantSupplierCode.variant` -> `Variant` (Cascade) -- correto, dependente

### 4.2 Cascades Ausentes (Warning)

- **Item -> Variant**: `Item.variant` usa o Prisma default (Restrict). Se um `Variant` for hard-deleted, os `Item` associados ficariam orfaos. **Atenuante**: O sistema usa soft-delete para `Variant`, entao nao ocorre hard-delete em operacao normal.

- **Item -> Bin**: `Item.bin` usa default (Restrict). **Atenuante**: Os bins sao soft-deleted e os itens sao desvinculados antes (`detachItemsFromBins`).

- **ItemMovement -> Item**: `ItemMovement.item` usa default (Restrict). Correto - movimentacoes nao devem ser deletadas em cascata.

- **PurchaseOrder -> Supplier**: `PurchaseOrder.supplier` usa default (Restrict). Correto - nao se deve perder historico de pedidos ao deletar um fornecedor.

### 4.3 Conflito entre Cascade e Soft Delete

Quando o sistema faz soft-delete de um `Product` (setando `deletedAt`), o Prisma **nao** propaga essa acao para `Variant`, `ProductCategory`, `ProductTag`, etc. Isso significa que:

- Variantes de um produto soft-deleted continuam com `deletedAt: null`
- O `DeleteProductUseCase` (linhas 13-27) apenas faz `productsRepository.delete()` sem tocar nas variantes filhas

**Cenario**: Um produto e soft-deleted, mas suas variantes continuam "ativas". Se alguem listar variantes, elas aparecerao como disponiveies mas sem produto pai. Os repositorios de variante filtram por `deletedAt: null` mas nao verificam se o produto pai tambem esta ativo.

**Recomendacao**: O `DeleteProductUseCase` deveria tambem soft-deletar as variantes associadas (e os itens, em cascata logica).

## 5. Race Conditions [FAIL]

### 5.1 Sequencial Code para Variantes - Race Condition Critica

**Arquivo**: `src/use-cases/stock/variants/create-variant.ts`, linhas 182-186

```typescript
const lastVariant = await this.variantsRepository.findLastByProductId(productId, input.tenantId);
const nextSeq = (lastVariant?.sequentialCode ?? 0) + 1;
```

Este padrao de "read-then-write" e vulneravel a race conditions. Se duas requisicoes concorrentes tentarem criar variantes para o mesmo produto:

1. Requisicao A le `lastVariant.sequentialCode = 3`, calcula `nextSeq = 4`
2. Requisicao B le `lastVariant.sequentialCode = 3`, calcula `nextSeq = 4`
3. Requisicao A cria variante com `sequentialCode = 4`, `fullCode = "001.001.0001.004"`
4. Requisicao B tenta criar variante com `sequentialCode = 4`, `fullCode = "001.001.0001.004"` -> **FALHA** por violacao de unique constraint no `fullCode`

**Mitigacao existente**: O `fullCode` tem `@unique`, entao a constraint do banco evita dados duplicados. Porem, a requisicao B recebe um erro 500 em vez de um retry automatico.

### 5.2 Sequencial Code para Itens - Mesma Race Condition

**Arquivo**: `src/use-cases/stock/items/register-item-entry.ts`, linhas 155-159

Mesmo padrao "read-then-write" para itens:

```typescript
const lastItem = await this.itemsRepository.findLastByVariantId(variantId, input.tenantId);
const nextSeq = (lastItem?.sequentialCode ?? 0) + 1;
```

### 5.3 Product.getNextSequentialCode - Seguro (mas com gap)

O `getNextSequentialCode` para `Product` e `Manufacturer` usa `SELECT nextval(...)` que e atomico no PostgreSQL e nao sofre de race condition. Porem, se a criacao do produto falhar apos obter o `nextval`, a sequence tera um gap permanente.

### 5.4 Volume Code Generation - Race Condition

**Arquivo**: `src/use-cases/stock/volumes/create-volume.ts`, linhas 31-56

O codigo do volume e gerado com `Date.now().toString().slice(-6)` + random. A verificacao de unicidade e feita via `findByCode`, mas entre a verificacao e a criacao pode haver uma requisicao concorrente que usa o mesmo codigo. **Atenuante**: A constraint `@@unique([code, tenantId])` no banco evita duplicacao, mas a requisicao recebera um erro 500.

## 7. Foreign Keys - Integridade Referencial [WARN]

### 7.1 VolumeItem sem tenantId

O modelo `VolumeItem` nao possui campo `tenantId`. A integridade e garantida indiretamente via `Volume.tenantId`, mas isso impede queries diretas filtradas por tenant na tabela `volume_items`.

### 7.2 VariantPriceHistory, VariantImage, UnitConversion sem tenantId

Estes modelos dependentes nao possuem `tenantId` proprio. A integridade multi-tenant e garantida via relacao com `Variant`, que tem `tenantId`. Isso e aceitavel para tabelas de juncao/historico, mas impede queries diretas filtradas por tenant.

### 7.3 Alert sem tenantId

O modelo `Alert` nao possui `tenantId`. Alertas sao filtrados por `userId`, mas isso dificulta operacoes administrativas por tenant.

### 7.4 ProductCategory sem tenantId

A tabela `product_categories` nao tem `tenantId`. A integridade e garantida via `Product` e `Category`, ambos com `tenantId`. Aceitavel para tabela de juncao.

### 7.5 Soft Delete de Product nao propaga para filhos

Conforme descrito em 4.3, o `DeleteProductUseCase` nao soft-deleta variantes, itens, ou registros de `ProductCategory`/`ProductTag` associados. Isso pode criar registros orfaos logicamente (variantes "ativas" cujo produto pai esta deletado).

## 9. Seed Data Idempotente [PASS]

O seed (`prisma/seed.ts`) usa padroes de upsert e verificacao de existencia antes de criar dados. O `release.sh` executa o seed automaticamente em cada deploy, o que requer idempotencia. Conforme documentado em MEMORY.md, o seed esta automatizado e funcional desde Fevereiro 2026.

## Recomendacoes Priorizadas

### Prioridade 1 - Critica (devem ser resolvidas antes do proximo deploy)

1. **Adicionar transacao a `RegisterItemEntryUseCase`**: Injetar `TransactionManager` e envolver `itemsRepository.create()` + `itemMovementsRepository.create()` em `transactionManager.run()`.

2. **Adicionar transacao a `RegisterItemExitUseCase`**: Mesma abordagem - envolver `itemsRepository.save()` + `itemMovementsRepository.create()`.

3. **Adicionar transacao a `TransferItemUseCase`**: Envolver `itemsRepository.save()` + `itemMovementsRepository.create()`.

4. **Adicionar transacao a `ConfigureZoneStructureUseCase`**: Todas as 7 operacoes de escrita devem executar dentro de uma unica transacao.

5. **Adicionar transacao a `DeleteZoneUseCase`**: Todas as operacoes de detach + delete + movement devem ser atomicas.

### Prioridade 2 - Alta

6. **Resolver race condition nos sequentialCode de Variant e Item**: Substituir o padrao "read-then-write" por:
   - Opcao A: Usar `SELECT ... FOR UPDATE` + transacao (pessimistic lock)
   - Opcao B: Implementar retry com backoff exponencial em caso de violacao de unique constraint
   - Opcao C: Usar sequence do PostgreSQL por tenant (mais complexo, mais correto)

7. **Garantir `BatchTransferItemsUseCase` sempre use transacao**: Alterar o `transactionManager` de opcional (`?`) para obrigatorio no construtor.

8. **Adicionar soft-delete em cascata logica no `DeleteProductUseCase`**: Ao soft-deletar um produto, tambem soft-deletar todas as variantes associadas.

### Prioridade 3 - Media

9. **Adicionar `tenantId` a clausula `where` dos metodos update/save/delete**: Todos os repositorios Prisma devem incluir `tenantId` na clausula `where` para operacoes de mutacao, como defesa em profundidade.

10. **Adicionar `tenantId` ao modelo `VolumeItem`**: Ou, alternativamente, adicionar filtro de tenant via join nas queries de `VolumeItem`.

11. **Tornar `Template.code` unique por tenant**: Alterar de `@unique` para `@@unique([code, tenantId])`.

12. **Adicionar transacao a `ReorderCategoriesUseCase`**: Envolver todos os updates em uma unica transacao para garantir consistencia na ordem de exibicao.

### Prioridade 4 - Baixa

13. **Tornar `Product.getNextSequentialCode` scoped por tenant**: Usar uma tabela auxiliar de sequences por tenant em vez da sequence global do PostgreSQL.

14. **Adicionar retry automatico para race conditions no `CreateVolumeUseCase`**: O `generateVolumeCode` ja tem retry, mas deveria capturar o erro de unique constraint do banco e fazer retry em vez de falhar.
